Part 1  ·  What Is a Work Breakdown Structure? (12 min)

Breaking a Robot Build Into Manageable Pieces

A Work Breakdown Structure (WBS) is a hierarchical decomposition of a project into progressively smaller pieces of work. At the top is the project goal. Below it are major phases. Below each phase are tasks. Below each task are subtasks. Every leaf of the tree is something one person can own, estimate, and complete.

The WBS answers: What needs to happen? A Gantt chart answers: When does each thing happen, and in what order? Tasks often depend on other tasks — you can't wire motors before the motors are mounted. These dependencies create a sequence. The longest chain of dependent tasks is called the critical path: if any task on the critical path slips, the whole project is late.

Some tasks can happen in parallel — two people working on different subsystems simultaneously. Identifying which tasks are independent (parallelizable) and which are dependent (sequential) is one of the highest-leverage decisions a project manager makes. In FTC, teams that build in parallel — different members working on drivetrain and mechanism simultaneously — consistently finish earlier and with more time to iterate.

I Do Watch the teacher model decomposing one phase of the robot build into a WBS

Starting at the project level: "The project is 'Build a competition-ready robot.' That's too big to work with. I decompose it into 5 phases: (1) Engineering Foundations, (2) Hardware Build, (3) Electronics & Wiring, (4) Software, (5) Integration & Testing. Each phase has a milestone — a moment when I can confirm that phase is complete before the next begins."

Decomposing one phase — Hardware Build: "Tasks: Design drivetrain, Build chassis, Design mechanism, Build mechanism, Mount mechanism to chassis. Each task has subtasks. 'Build chassis' = (a) Cut extrusions to length, (b) Assemble frame, (c) Verify squareness, (d) Attach wheels and axles. Each subtask has an owner and an estimate: (a) Parts Manager — 15 min, (b) Build Lead — 30 min, (c) Quality Lead — 10 min, (d) Build Lead — 20 min."

Identifying the critical path: "Design drivetrain must happen before Build chassis. Build chassis must happen before Mount mechanism. These are sequential — if Design drivetrain takes longer than expected, everything shifts. But: while the chassis is being built, the Documentation Lead can be preparing the wiring diagram in parallel. That's not on the critical path because it doesn't block chassis construction."

Buffer time — the thing teams always skip: "Every estimate is wrong. Budget 20% extra on every task. If a task is estimated at 1 class period, plan for 1.2. This buffer is not sloppiness — it's engineering reality. The teams that schedule buffer time into their Gantt chart are the teams that have time to iterate before competitions."

Part 2  ·  Practice Activity: Chassis Mini-Project Gantt (remainder of this class + all of next class  ≈ 105 min total)

Plan Before You Build — A Micro-Scale Gantt in Minutes, Not Classes

The course-level Gantt you just completed uses classes as its time unit. Real engineering projects often need finer resolution — when you have 105 minutes to design and build something, you need a plan in minutes, not weeks. This mini-project gives you exactly that: you will plan a chassis build at minute-level resolution, execute the plan, and then compare what actually happened to what you predicted.

The chassis is a structural-elements-only build — no motors, no motor mounts. Your team will choose between REV 15mm extrusion and REV C Channel as your primary structural material and design a frame that could accept all the robot's subsystems later. The chassis is the integration layer: every other module the team builds this semester will connect to it. That means getting it square, sturdy, and well-documented matters as much as getting it built.

You will work through all seven EDP stages used throughout this course: Define → Research → Ideate → Prototype → Test & Measure → Analyse → Iterate. Your Gantt chart allocates minutes to each stage before you start. At the end of each stage you mark the actual time used. The gap between planned and actual is data — and data about your own planning is the most useful kind.

Part 3  ·  Milestones & Stand-Up Practice (10 min)

Checkpoints That Confirm a Phase Is Truly Done

A milestone is a specific, verifiable event that confirms a phase is complete. It is not "mostly done" or "about done" — it is either complete or not. Good milestones have a clear owner, a clear test, and a clear date. "Chassis built" is not a milestone. "Chassis passes squareness test (diagonals within 2mm) — Quality Lead signed off — Class 11" is a milestone.

Extension / Exploration

Research PERT (Program Evaluation Review Technique) — a probabilistic scheduling tool used on the Apollo space program. A PERT chart shows three estimates for each task: optimistic, most likely, and pessimistic. Apply a basic PERT analysis to the EDP stage that had your largest planned vs. actual variance today. Using your actual data as the reference: estimate O (best case, if everything had gone perfectly), M (most likely, matching your actual time), and P (worst case). Calculate E = (O + 4M + P) ÷ 6. How does the PERT estimate compare to your original plan? Write 2 sentences explaining what this reveals about using historical data to improve future estimates.
